01 サービスコンセプト 確定

サービス名： スナログ（snacklog）

一言コンセプト：
「初心者・女性でも安心して行ける」赤坂特化のスナック検索・口コミ・来店きっかけサイト。将来的には全国展開を想定した拡張可能な構造。

重要な定義：
「予約サイト」ではなく 「来店きっかけサイト」 と定義する。スナックの来店行動は流れ来店が主であり、予約コンバージョンは計測・期待できない。

参照モデル

  • 食べログ（網羅性・透明性）
  • エアビー（市場原理）
  • Airレジ / TableCheck（経営管理）
  • バイトル / ランサーズ（求人）

3者が使うプラットフォーム

来店ユーザー ／ 店舗オーナー・ママ ／ 求職者（働き手）

#タイプ主な課題サイトへの期待潜在規模
①スナック初心者（30〜40代男性）雰囲気・料金不透明 / 1人で入るのが不安初心者歓迎タグ・明朗会計・写真で安心50〜160万人
②出張旅・地方営業マン（30〜50代男性）知らない街での店探しが難しい現在地検索 / 1人歓迎 / 今夜行ける店30〜60万人
③スナック好き常連志向層（40〜60代男性）自分に合う新しい店が見つからないママ図鑑形式で雰囲気比較30〜90万人
④女性客・カップル（20〜40代）女性でも入りやすいか不安 / 映え重視女性歓迎タグ / 写真・内装・BGM情報16〜60万人
共通の心理的ハードル：雰囲気・料金の不透明さ。「予約」より「今夜ふらっと行く」即時行動が多い。

06 各ページ詳細仕様 作成中

TOP（ホーム） / MVP・最優先

目的

「今夜、赤坂でスナックを探している人」が直感的に検索を始められること。ログイン会員にはフレンドのチェックイン情報も表示。

グローバルナビゲーション（ヘッダー・常時表示）

  • ロゴ（クリックでTOPへ）
  • 検索バー
  • ログイン / 会員登録ボタン（未ログイン時）／ マイページアイコン（ログイン時）
  • 掲載依頼ボタン
  • ハンバーガーメニュー（右上：求人・プラン比較等を格納）

主要エリア構成

  • ヒーローエリア：大型検索バー（エリア・キーワード入力）
  • クイックフィルター：初めての方 / 一人飲み / カラオケあり / 女性歓迎
  • 今盛り上がってる店（リアルタイムランキング）：チェックイン数ベースのランキング表示。会員のチェックインにより動的更新
  • 今夜行けるスナック：即時来店フラグ付き店舗の一覧
  • フレンドの動向（ログイン会員のみ表示）：フレンドが現在チェックイン中の店舗をお知らせ
  • プレミアム掲載店スポットライト：プレミアムプラン店舗の優先表示
  • 人気エリアリンク：赤坂見附・溜池山王

チェックイン機能（会員ログイン時）

  • TOPページまたは店舗詳細ページからワンタップでチェックイン実行
  • GPS位置情報と連動（任意）
  • チェックイン中は「今ここにいます」バッジを店舗ページに表示
  • フレンドに通知が届く（通知設定でON/OFF可能）
  • チェックアウトは手動 or 翌日自動解除
  • 訪問履歴としてマイページに自動記録

CV導線

検索実行 → 店舗詳細 → 電話 / SNS / 予約 / チェックイン

フレンド管理 /mypage/friends MVP 新規追加

目的

フレンド関係の管理と通知設定を一元管理する専用ページ。

主要機能

  • フレンド一覧：繋がっているフレンドの一覧表示・現在のチェックイン状況も表示
  • フレンド申請管理：受信した申請への承認 / 拒否、送信した申請のキャンセル
  • フレンド検索：ニックネームやIDで他の会員を検索して申請
  • 通知設定：フレンドがチェックインしたときの通知 ON/OFF を個別に設定可能
  • フレンドの削除：フレンド関係の解除

フレンド機能の仕様

  • 相互承認型：申請 → 相手が承認して初めてフレンド成立
  • 一方的なフォローは不可（SNSとは異なるクローズドな設計）
  • チェックイン通知は受信側が個別にON/OFF設定可能
  • 通知内容：「○○さんが△△スナックにチェックインしました」

チェックイン情報の公開範囲設定

  • デフォルト：フレンドのみ公開（申請・承認したフレンドにのみチェックイン状況が見える）
  • 選択肢：「フレンドのみ」／「非公開（誰にも見せない）」をマイページで切り替え可能
  • 「全体公開」は原則非対応（スナックという業態のプライバシー配慮から）
  • TOPページの「今盛り上がってる店」ランキングはチェックイン数の集計のみ表示（個人は特定不可）

店舗管理ダッシュボード /manage MVP

目的

店舗オーナーが掲載情報を管理し、集客効果を確認する。

掲載申請〜掲載開始フロー

  • ① 店舗オーナーが掲載申請フォームから情報を送信
  • ② スナログ運営が審査（簡易確認：電話番号・屋号の照合）
  • ③ 承認後、店舗管理ダッシュボードへのアクセス権を付与
  • ④ オーナー自身が情報を入力・公開設定

初期フェーズ（Phase1）は運営が手動対応。Phase2以降は自動審査フローを検討（論点13-5参照）。

機能一覧

  • 店舗情報編集（写真・基本情報・料金・SNSリンク・今夜行けるフラグ）
  • CVレポート（電話・SNSタップ数・MAP起動数）
  • 予約管理（団体予約の確認・承認 ※プレミアム）
  • 求人管理・応募者確認（※プレミアム）
  • BIダッシュボード：売上・来客推移（スタンダード以上）
  • 顧客管理（プレミアム）：来店歴のあるフレンドのグルーピング・グループ別メッセージ送信
  • ボトルキープ管理：登録名・銘柄・期限・残量の管理 → 会員マイページと連携
  • プッシュ通知送信（スタンダード以上）：チェックイン会員や既存顧客への通知配信

事業承継・店舗買収相談 /ma Phase4 新規追加

目的

後継者不在や廃業を検討しているスナックオーナーと、店舗取得・開業を希望する買い手をマッチングする。スナックの検索・発見プラットフォームとしての信頼性・ネットワークを活かした事業承継支援サービス。

背景・機会

  • スナック業界では後継者不在による廃業が増加しており、事業承継ニーズが高い
  • スナログが掲載する店舗オーナーとの信頼関係・データ（来客数・売上推移等）を活用可能
  • 買い手（新規開業者・投資家）向けの物件情報・経営データ提供が差別化になる

主要機能

  • 売却相談フォーム：オーナーが売却・廃業・承継を相談できる窓口（プレミアムプラン店舗向け）
  • 買収物件一覧：条件・エリア・売上規模などで絞り込み検索（会員限定）
  • マッチング仲介：スナログが間に入り、売り手・買い手の条件調整をサポート
  • BI連携：プレミアムプランのBIダッシュボードデータを買収判断材料として提供

収益モデル

  • 成功報酬型：マッチング成立1件につき¥1,000,000（試算用単価）
  • 年10件成立目標（Phase4以降）→ 年間収益¥1,000万規模
  • 相談登録・情報掲載は無料（成立時のみ課金）

法的考慮事項

不動産仲介・M&A仲介に関する法規制の確認が必要。原田さんへの相談事項として論点リストに追記予定。

通知種別方式対象優先度

技術方式の選択肢

  • アプリ内通知（バッジ）：ウェブプッシュAPIまたはWebSocket。実装コスト低。ブラウザを開いていないと届かない
  • プッシュ通知（PWA）：Progressive Web App化でスマホへのプッシュ通知が可能。ユーザーの許可が必要
  • メール通知：SendGrid / Resend等を活用。確実に届くが即時性は低い

MVP段階はアプリ内通知（バッジ表示）とメール通知の組み合わせを推奨。PWA対応はPhase2以降で検討。担当：りょうちゃん。

13-4 モデレーション設計（口コミ・コンテンツ管理）

口コミの審査フロー

  • Phase1：事後審査（投稿後に公開→問題があれば対応）。運営工数を最小化
  • Phase2以降：AIフィルタリング（不適切ワード・スパム検出）→ 問題なければ即時公開
  • 店舗オーナーは「この口コミに問題を報告」ができ、運営が確認・判断

不適切コンテンツの定義と対応方針

種別対応
誹謗中傷・個人攻撃即時非公開 → ユーザーへ警告 → 繰り返しでアカウント停止

広告・スパムAIフィルタで自動排除

13-5 店舗オーナーの本人確認・掲載審査フロー

Phase1（手動審査）

  • 掲載申請フォームから屋号・住所・電話番号・担当者名・連絡先メールを送信
  • 運営が電話での折り返し確認（本人確認）
  • 確認完了後、店舗管理画面へのアクセス権を発行（最大3営業日）

Phase2以降（半自動化）

  • Google Business Profileとの連携で本人確認を代替（GBP所有者確認済みであれば審査省略）
  • SMS認証（電話番号宛てにワンタイムコード）を追加
  • 書類アップロード（営業許可証・風俗営業許可証等）の必要有無は原田さん確認後に判断
