01 サービスコンセプト 確定

サービス名： スナログ（snacklog）

一言コンセプト：
「初心者・女性でも安心して行ける」赤坂特化のスナック検索・口コミ・来店きっかけサイト。将来的には全国展開を想定した拡張可能な構造。

重要な定義：
「予約サイト」ではなく 「来店きっかけサイト」 と定義する。スナックの来店行動は流れ来店が主であり、予約コンバージョンは計測・期待できない。

参照モデル

  • 食べログ（網羅性・透明性）
  • エアビー（市場原理）
  • Airレジ / TableCheck（経営管理）
  • バイトル / ランサーズ（求人）

3者が使うプラットフォーム

来店ユーザー ／ 店舗オーナー・ママ ／ 求職者（働き手）

02 ターゲットユーザー（ペルソナ） 確定

#タイプ主な課題サイトへの期待潜在規模
①スナック初心者（30〜40代男性）雰囲気・料金不透明 / 1人で入るのが不安初心者歓迎タグ・明朗会計・写真で安心50〜160万人
②出張旅・地方営業マン（30〜50代男性）知らない街での店探しが難しい現在地検索 / 1人歓迎 / 今夜行ける店30〜60万人
③スナック好き常連志向層（40〜60代男性）自分に合う新しい店が見つからないママ図鑑形式で雰囲気比較30〜90万人
④女性客・カップル（20〜40代）女性でも入りやすいか不安 / 映え重視女性歓迎タグ / 写真・内装・BGM情報16〜60万人
共通の心理的ハードル：雰囲気・料金の不透明さ。「予約」より「今夜ふらっと行く」即時行動が多い。

06 各ページ詳細仕様 作成中

TOP（ホーム） / MVP・最優先

目的

「今夜、赤坂でスナックを探している人」が直感的に検索を始められること。ログイン会員にはフレンドのチェックイン情報も表示。

グローバルナビゲーション（ヘッダー・常時表示）

  • ロゴ（クリックでTOPへ）
  • 検索バー
  • ログイン / 会員登録ボタン（未ログイン時）／ マイページアイコン（ログイン時）
  • 掲載依頼ボタン
  • ハンバーガーメニュー（右上：求人・プラン比較等を格納）

主要エリア構成

  • ヒーローエリア：大型検索バー（エリア・キーワード入力）
  • クイックフィルター：初めての方 / 一人飲み / カラオケあり / 女性歓迎
  • 今盛り上がってる店（リアルタイムランキング）：チェックイン数ベースのランキング表示。会員のチェックインにより動的更新
  • 今夜行けるスナック：即時来店フラグ付き店舗の一覧
  • フレンドの動向（ログイン会員のみ表示）：フレンドが現在チェックイン中の店舗をお知らせ
  • プレミアム掲載店スポットライト：プレミアムプラン店舗の優先表示
  • 人気エリアリンク：赤坂見附・溜池山王

チェックイン機能（会員ログイン時）

  • TOPページまたは店舗詳細ページからワンタップでチェックイン実行
  • GPS位置情報と連動（任意）
  • チェックイン中は「今ここにいます」バッジを店舗ページに表示
  • フレンドに通知が届く（通知設定でON/OFF可能）
  • チェックアウトは手動 or 翌日自動解除
  • 訪問履歴としてマイページに自動記録

CV導線

検索実行 → 店舗詳細 → 電話 / SNS / 予約 / チェックイン

店舗検索（一覧） /search MVP・最優先

目的

ユーザーが条件を絞り込み、自分に合う店を発見する。

絞り込み条件

  • エリア（赤坂見附 / 溜池山王 / 赤坂）
  • 現在地GPS検索（周辺◯km以内）
  • 予算（〜¥5,000 / 〜¥10,000 / ¥10,000〜）
  • 設備タグ（カラオケ / ボトルキープ / カード可）
  • 雰囲気タグ（初心者歓迎 / 女性歓迎 / 一人歓迎）
  • 今夜行けるフラグ

店舗カード

写真・店名・エリア・料金目安・口コミ評価・タグ・今夜行けるバッジ・現在のチェックイン人数バッジ

表示順

プレミアム上位固定 → スタンダード（マップピン）→ ベーシック

店舗詳細 /shops/:id MVP・最優先

目的

安心して来店できるよう十分な情報を提供し、CTAへ繋げる。

主要要素

  • 写真ギャラリー（3枚 / 10枚 / 無制限＋動画：プランによる）
  • 基本情報（住所・営業時間・定休日・地図）
  • 料金システム（セット料金 / チャージ / ボトルキープ）
  • ママ紹介（写真＋コメント）※スタンダード以上
  • タグ一覧（雰囲気・設備）
  • 今チェックインしている会員数バッジ
  • 口コミ一覧・投稿フォーム
  • 類似店舗（横スライダー）
  • 求人情報（掲載している場合のみ）

CTA（固定ボタン）

  • 📞 電話する（常時表示）
  • 💬 SNSで問い合わせ（登録済みSNSのみ表示）
  • 📅 予約する（団体向け・予約URL登録時のみ）
  • 📍 チェックイン（会員ログイン時のみ表示）

マイページ /mypage MVP

目的

会員ユーザーの活動履歴管理とフレンド・通知の設定拠点。

主要機能

  • 訪問履歴（チェックイン履歴・訪問した店舗一覧）
  • お気に入り店舗一覧
  • フレンドの動向（現在チェックイン中のフレンドと店舗）
  • 会員プランの確認・変更
  • プロフィール編集（ニックネーム・アイコン）

フレンド管理 /mypage/friends MVP 新規追加

目的

フレンド関係の管理と通知設定を一元管理する専用ページ。

主要機能

  • フレンド一覧：繋がっているフレンドの一覧表示・現在のチェックイン状況も表示
  • フレンド申請管理：受信した申請への承認 / 拒否、送信した申請のキャンセル
  • フレンド検索：ニックネームやIDで他の会員を検索して申請
  • 通知設定：フレンドがチェックインしたときの通知 ON/OFF を個別に設定可能
  • フレンドの削除：フレンド関係の解除

フレンド機能の仕様

  • 相互承認型：申請 → 相手が承認して初めてフレンド成立
  • 一方的なフォローは不可（SNSとは異なるクローズドな設計）
  • チェックイン通知は受信側が個別にON/OFF設定可能
  • 通知内容：「○○さんが△△スナックにチェックインしました」

店舗管理ダッシュボード /manage MVP

目的

店舗オーナーが掲載情報を管理し、集客効果を確認する。

機能一覧

  • 店舗情報編集（写真・基本情報・料金・SNSリンク・今夜行けるフラグ）
  • CVレポート（電話・SNSタップ数・MAP起動数 ※スタンダード以上）
  • 予約管理（団体予約の確認・承認 ※プレミアム）
  • 求人管理・応募者確認（※プレミアム）
  • BIダッシュボード：売上・来客推移（※プレミアム）

口コミ投稿 /shops/:id/review Phase2

目的

来店証明コンテンツを蓄積し、次のユーザーの意思決定を支援する。

入力項目

  • 総合評価（★1〜5）
  • 雰囲気・料金・ママの対応（各★）
  • テキストコメント（任意）
  • 来店日・人数・利用シーン

求人（一覧・詳細・応募） /jobs Phase2

目的

スナックで働きたい人と店舗をマッチングする。

一覧の要素

店名・エリア・時給・シフト・応募条件・写真

プレミアムプランの店舗のみ掲載（検討中：論点A-3）

プラン比較ページ /pricing Phase2

目的

店舗オーナー・ユーザーが自分に合ったプランを選べるよう、機能・価格を明確に比較提示する。

主要要素

  • 店舗向け3プラン横並び比較表（ベーシック / スタンダード / プレミアム）
  • ユーザー向け2プラン比較表（ベーシック会員 / プレミアム会員）
  • 機能マトリクス（● / — で一覧）
  • 各プランCTAボタン
  • よくある質問（FAQ）

10 UI / UX方針 確定

  • スマホファースト（レスポンシブ）
  • TOPは来店ユーザー体験を最優先
  • 全国展開を前提とした拡張可能な構造
  • 店舗・求人向け入口はメニューに集約

初心者でも入れるよう、料金の透明性・写真・ママ紹介を前面に出す。チェックイン・フレンド機能で「今夜誰かと飲みに行く」自然な動線を作る。

11 競合分析・差別化 確定

競合弱点
スナックガイド情報が古くUXが弱い
ナイトスタイルキャバクラ中心でスナック専門性なし
ホットペッパーグルメスナックカテゴリが弱い
Googleマップ口コミ中心で体系化されていない
スナックナビ月6万セッション・SNSとの体系化なし

差別化の核： 「信頼性の高い情報 ＋ 来店きっかけの導線 ＋ チェックイン・フレンドによる"今夜一緒に飲みに行く"体験の創出」
